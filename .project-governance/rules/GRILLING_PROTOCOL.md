# Grilling Protocol

本协议内置 `grill-me` 的核心行为，不依赖外部 skill。

## 何时追问

必须追问：新项目/功能/版本/需求；需求、交互、架构、技术栈、数据模型、API、部署变化；验收不通过后的修改方向；多路线反馈；文档、代码、用户口述冲突；任何影响阶段、SSOT、版本骨架或验收标准的决定；新概念引入前；流程改动；版本启停；阶段回归。

通常可豁免：明确 bug 修复、测试补充、格式/拼写、无行为变化重命名、不改变行为的依赖升级、用户已完整说明的小任务。豁免时说明"不影响 PRD、架构、API、流程和版本"。

## 怎么追问

- 一次只问一个关键问题，串行推进；用户回答前不得抛出下一个问题。
- 每个问题都必须给出 agent 的推荐答案，并附上基于当前项目上下文的理由，让用户可以直接确认或否决。
- 先解决上游决策，再问下游细节；待问清单按重要性排序，每次只暴露最关键的那一个给用户。
- 能通过读项目文件回答的，先读文件，不问用户。
- 用户明确确认前，不得把争议假设写入 PRD、架构、API、流程文件或代码。
- 与用户交流时：优先使用 `ssot/GLOSSARY.md` 中 `User Term` 列的词；不堆砌后台文档变量、字段名或专有名词；必须引用具体文件或术语时，先用一句白话解释它在当前语境下指什么；引入新概念前必须先登记对照表条目。
- 把用户的话写入 SSOT、决策、流程文件前，必须查 `ssot/GLOSSARY.md` 把口语词反翻译为 `Doc Term` 列的标准词，避免同一概念在文档中出现多种叫法。用户的话匹配到多个术语或匹配不到任何条目时，必须触发追问确认。

## 什么算确认

只有用户明确表达"确认""同意""按这个方案做""验收通过""进入下一阶段"或等价授权，才算确认。沉默、模糊赞同、agent 自行判断都不算确认。

## 追问后写哪里

- 需求/交互：`ssot/PRD.md`
- 架构/数据模型/部署：`ssot/ARCHITECTURE.md`
- 接口：`ssot/API_CONTRACT.md`
- 术语/对照：`ssot/GLOSSARY.md`
- 骨架阶段状态 / 版本骨架 / 回归记录 / Backlog / Stage Skips：`ssot/PROJECT_STATE.md`
- 实际开发段内部流程定义：`.project-governance/processes/active.md`
- 流程库写入：`~/.claude/process-library/<name>.md`（用户本机私有，沉淀时操作）
- 长期原因和权衡：`.project-governance/decisions/NNNN-title.md` + `.project-governance/decisions/INDEX.md`
