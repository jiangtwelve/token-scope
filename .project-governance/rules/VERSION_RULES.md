# Version Rules

本文件描述版本骨架、回归、Backlog 与流程沉淀触发规则。版本相关的状态记录在 `ssot/PROJECT_STATE.md`。

## 版本结构

- 项目状态文件下设 `Versions` 段，每个版本一份独立的骨架四段（参见 `rules/DEVELOPMENT_PROCESS.md`）。
- 历史版本走完四段后整体标记 `Released`，结构与状态不再修改；新版本以追加形式记录在其后。
- 同一版本内允许阶段回归（见下文），跨版本通过 `Versions` 段的追加体现，不允许通过修改旧版本骨架来表达。

## 版本启动与结束

- 当前版本第四段（验收测试）`Confirmed` 即具备结束条件。agent 必须主动询问用户："是否开新版本？版本号？范围？"按追问协议一次一问。
- 用户回答"开新版本"后，agent 追加新版本骨架（四段状态 `Not Started`），并自动扫描 `Backlog` 段，把 `target_version` 命中新版本号或留空且建议进入新版本的条目逐条提示用户。
- 当前版本未完成（任何一段未 `Confirmed`）时，禁止启动新版本。用户若要求强行升版本，agent 必须按下述"硬升应对"处理。

## 硬升应对

用户在当前版本未结束时要求开新版本：

1. agent 必须拒绝直接升版本，并向用户说明原因（当前版本仍在进行）。
2. agent 引导用户判断：是否把对应功能并入当前版本继续开发；或者标记为待开发并放入 `Backlog`。
3. 用户的最终选择写入决策记录，并相应更新 `ssot/PRD.md` / `Backlog`。

## 版本号规则

- 版本号由用户拍板，不要求连续（1.0 → 1.8 → 2.4 都允许）。
- agent 必须在用户给出的版本号违反数字大小直觉（逆序、间隔异常、与已有版本号冲突）时按追问协议提醒，但不拒绝；用户坚持时写入决策记录。
- 若用户在需求讨论阶段给出版本规划，agent 应主动建议规划版本号的命名约定，避免后期混乱。

## 阶段回归

- 状态字段：`Not Started` / `In Progress` / `Confirmed` / `Blocked: regressed to <stage_id>`。
- 回归动作：
  1. 把目标段状态从 `Confirmed` 改回 `In Progress`；
  2. 把当前段状态改为 `Blocked: regressed to <stage_id>`；
  3. 在 `Stage Regressions` 段记录：从哪段回到哪段、原因、用户确认。
- 回归只在当前版本骨架内发生；不允许通过回归动作影响旧版本的状态。
- 回归与跳过是不同动作：跳过是用户对 `optional: true` 阶段的一次性豁免；回归是把已 `Confirmed` 的段重新打开。跳过规则见 `rules/DEVELOPMENT_PROCESS.md` 的"跳过记录"段。

## Backlog

- `Backlog` 段记录"被推迟但未明确进入某版本"的功能。
- 每条记录字段：描述、来源（来自哪个版本的哪个阶段被推迟）、目标版本（可空）、原因、用户确认时间。
- 触发场景：
  - 实际开发中用户决定推迟某功能且未指定目标版本时，agent 必须按追问协议确认是否入 backlog 还是指定目标版本。
  - 需求讨论阶段 agent 主动建议把某项功能推迟时，确认后入 backlog。
- 所有规划版本都完成后若 backlog 仍有未消化条目，agent 主动引导用户新开版本承接，按追问协议确认。

## 沉淀触发

流程库写入的两个触发点：

1. **首版（v1.0）第四段 `Confirmed` 时**：agent 强制询问"是否把本项目的开发流程沉淀进流程库"。
2. **后续版本若改过流程并完整走完一轮（第四段 `Confirmed`）时**：agent 询问"是否把改过的流程入库（替换旧版 / 新增为一份新流程 / 不入库）"。

未改过流程的常规版本不再追问。

沉淀询问必须附 agent 建议与理由。用户决策结果（入库 / 替换 / 新增 / 不入库）一律写入决策记录。沉淀写入操作落到 `~/.claude/process-library/`，由 agent 与用户共同填写流程模板字段（其中 `lessons_learned` 由用户口述、agent 整理）。
